# Feature Research

**Domain:** Local-first university exam review desktop/Android app (Flutter)
**Researched:** 2025-01
**Confidence:** MEDIUM-HIGH

> **域定位说明：** RedClass 是"自带题库"型刷题工具——核心差异化在于"导入 docx/pdf → 本地 LLM 解析 → 立刻刷题"的零摩擦链路。比较对象包括 Anki（间隔重复，手工制卡）、Quizlet（云端题库）、小猿题库/粉笔（内置题库，无导入路径）、考研帮（垂直社区+题库）。RedClass 与所有这些在"用户是否拥有题库"这一维度上根本不同。

## Feature Landscape

### Table Stakes (Users Expect These)

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| `.docx` 文件导入 | 老师发放题库的最常见格式；缺失 = 无法覆盖 70%+ 场景 | MEDIUM | `docx` 包解压 + 段落抽取，UI 选文件用 `file_picker` |
| `.pdf` 文件导入 | 第二常见格式；扫描版/文字版两种路径 | MEDIUM-HIGH | 文字版用 `pdfx`/`syncfusion_flutter_pdf`；扫描版需 OCR（v1 范围见下） |
| 本地 LLM 解析 | 核心承诺；纯正则在格式浮动时崩 | HIGH | 通过 `llama.cpp`/GGUF 走 Dart FFI 或平台 channel；首启下载模型 |
| 解析进度 + 失败原因 | 用户在 30s+ 等待中无反馈会以为卡死 | LOW | 进度条 + 阶段提示（读取/分段/推理/写库）+ Toast 报错 |
| SQLite 持久化 | 数据不丢=基本信任 | LOW | `drift` 或 `sqflite`；表：题库、题目、答题日志、错题本、收藏 |
| 单选题支持 | 题库 80%+ 是单选 | LOW | radio 组件 + 提交时唯一正确答案 |
| 多选题支持（全部选对得分） | 第二高频题型 | LOW | checkbox + 必须完全匹配才得分；少选/多选均记错 |
| **乱序抽题** 模式 | 三模式中的基线 | LOW | 全题库随机抽 N 题；即时判分 |
| 错题自动入"错题本" | 模式间联动的核心 | LOW | 答题日志写入 + ledger 状态机（NEW / IN_LEDGER / MASTERED） |
| **错题复习** 模式 | 闭环复习的核心 | LOW | 仅从错题本抽题；答对 → MASTERED 并移出 ledger |
| **错题抽查** 模式 | 快速自测，排除已掌握 | LOW | 从 ledger 抽 N 题，filter `state != MASTERED` |
| 收藏 / 取消收藏单题 | 用户基础操作 | LOW | 题目表上加 `bookmarked` 布尔位 |
| 收藏夹查看与复习入口 | 收藏必须有去处 | LOW | 列表 + 复用"乱序抽题"UI 但限定题目集 |
| 答题统计（正确率/总数/按模式） | 用户判断复习效果的基本依据 | LOW | 从 `answer_log` 聚合，首页展示 3-4 个数字 |
| 启动可恢复上次状态 | 桌面 App 的基本尊严 | LOW | 保存当前题库/模式/题号；启动自动回到现场 |
| 退出/关闭时数据不丢 | 桌面 App 常见痛点 | LOW | 每次作答 commit 到 SQLite；关窗无确认弹窗 |
| Windows `.exe` 打包 | 主目标平台 | LOW | `flutter build windows`；便携式可考虑单文件 |
| Android `.apk` 打包 | 移动端基本交付 | LOW | `flutter build apk`；ARM64 必含，x86_64 可选 |

### Differentiators (Competitive Advantage)

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| **本地小 LLM 解析 docx/pdf** | 唯一能让"老师发的题库"零摩擦入库的能力；Anki/Quizlet 都没有这条路径 | HIGH | 选 0.5B–3B 量级的指令微调模型（Qwen2.5-1.5B-Instruct / Phi-3.5-mini / Gemma-2-2B），GGUF 量化 4-bit |
| 解析结果可预览+逐题编辑/删除 | LLM 解析不可能 100% 准；用户必须能修正 | MEDIUM | 解析完成后弹预览页，错误标记；`★` 重要 — 详见下"Anti-Features"对手工编辑的讨论 |
| 跨平台单代码库（Windows+Android） | 老师/学生/通勤/图书馆 任意场景都能用 | LOW（已有） | Flutter 已承诺；不维护两套是结构性优势 |
| 离线优先 / 零账号 / 隐私 | 与"国内刷题 App"形成鲜明差异；适合敏感题库（学校内部） | LOW | 无网络权限或仅模型下载；无后端 |
| 三模式错题本互通 | 比单一"错题本"更精细：复习 vs 抽查场景分开 | LOW | 状态机简单，UX 价值高 |
| 错题抽查排除已掌握 | 把"已掌握"显式化，给用户掌控感 | LOW | 一个 SQL 条件；UX 表达力强 |
| 答题用时记录 | 数据便宜但洞察强（慢答 = 薄弱） | LOW | 题目展示时启动 `Stopwatch`，提交时记录 |
| 题目来源（题库）切换与独立统计 | 多门课共用一个 App，统计必须按题库隔离 | LOW | 所有聚合查询都带 `bank_id` 过滤 |
| 解析任务可中断/后台运行 | 5 万字的题库可能要 10+ 分钟，必须能中断 | MEDIUM | Isolate + cancellation token；UI 提供停止按钮 |
| 解析结果批量确认 | 不让用户面对"1000 道题逐个审"的死路 | MEDIUM | 按章节/题型分组展示，定位到错题跳转修复 |

### Anti-Features (Commonly Requested, Often Problematic)

| Feature | Why Requested | Why Problematic | Alternative |
|---------|---------------|-----------------|-------------|
| **应用内富文本编辑器用于出题** | "我想补几道题进去" | 这是和"零摩擦"反向的能力：编辑器要支持图片、公式、公式对齐……一周时间起步；与 LLM 解析的核心差异化直接冲突 | **解析后预览编辑** 单题级别的简单表单（题干/选项/答案），不做富文本 |
| **填空题 / 判断题 / 简答题** | "大学考试也有这些题" | 填空：判分复杂（语义匹配？正则？字符集？）判断：技术上就是单选，多此一举简答：评分 = 调用 LLM，又多一层不可控 | v1 只做单选+多选；如未来要做，填空用精确字符串匹配 |
| **章节 / 知识点筛选** | "我想按第三章刷" | 老师题库未必有章节标签；强加会让用户手动标注，摩擦巨大 | 通过"题库"维度划分（一次导入 = 一门课），天然隔离 |
| **图片 / 公式 提取与渲染** | "我们高数题有图" | 提取：docx 图片内嵌，PDF 图片位置/OCR；渲染：LaTeX/MathJax Flutter 端支持差 | v1 留提示"该题含图片，请参看原文件"；v2 单独评估 |
| **云端同步 / 多设备** | "我想在手机接着做" | 加后端 = 加账号/加隐私/加服务器成本/加同步冲突；与"本地"核心承诺冲突 | 导出/导入 `.db` 文件（一键打包），不引入云 |
| **每日打卡 / 连续天数 / 成就** | "增加用户粘性" | 这是产品经理话术；对工具型应用是噪音；用户是来刷题的，不是来"维持 streak"的 | 不做。统计页给真实数据（正确率走势），不制造虚假的游戏化 |
| **社交 / 排行榜 / 班级对比** | "和同学 PK" | 与"个人闭源"和"无后端"双重冲突 | 不做 |
| **复杂统计仪表盘** | "数据可视化" | 折线图/雷达图/热力图在桌面端实现成本高，叙事价值低；学生不需要"刷了多少道"以外的洞察 | v1 三个数字：总题数、累计正确率、错题本剩余数；按需展开 |
| **题库分享市场** | "同学之间共享题库" | 涉及上传/审核/版权/分发，复杂度爆炸；和"自带题库"核心定位冲突 | 导出题库为标准文件，让用户自己用 U 盘/网盘传 |
| **多用户 / 角色（管理员/教师/学生）** | "老师能看学生进度" | 完全不同的产品；RedClass 是个人工具 | 不做 |
| **AI 出题 / 题目生成** | "LLM 反正能跑" | 出题质量不稳定、易编造错误知识点；评测困难 | 解析能力已足够覆盖核心场景 |
| **计时考试模式** | "我要模拟考" | 与"复习"定位不同；引入倒计时/防作弊后复杂度飙升 | 错题抽查已提供"短时高强度"自测 |
| **闪卡 / 间隔重复（Anki 模式）** | "Anki 证明有效" | 与"乱序抽题"算法假设不同；引入 SM-2/FSRS 需要题级记忆状态机 + 调度；和错题本状态机叠加，状态空间爆炸 | 用三模式 + 错题本已经覆盖 80% 收益 |

## Feature Dependencies

```
.docx / .pdf 导入
   └──requires──> 本地 LLM 推理
                       └──requires──> 模型文件已在本地（首次启动下载）
   └──requires──> 解析预览/编辑（保证入库质量）
                       └──requires──> 单题编辑表单

SQLite 持久化
   └──requires──> 数据模型定义（题库/题目/答题日志/错题本/收藏）
   └──requires──> 数据库路径在 Windows 与 Android 均可读写

乱序抽题
   └──requires──> 题库 + 题目已入库
   └──requires──> 单选/多选判分逻辑

错题本 ledger
   └──requires──> 答题日志写入
   └──requires──> 状态机：NEW → IN_LEDGER → MASTERED
   └──requires──> 错题复习模式
   └──requires──> 错题抽查模式（filter state != MASTERED）

收藏夹
   └──requires──> 题目标记字段
   └──enhances──> 乱序抽题（可限制为收藏范围）

答题统计
   └──requires──> 答题日志表
   └──enhances──> 错题本（统计"剩余错题数"）

题库管理（多门课）
   └──requires──> 题目按 bank_id 隔离
   └──conflicts──> 单题级"全局错题"（错题必须随题库走，不能跨题库合并统计）

LLM 解析后台运行
   └──conflicts──> UI 阻塞（必须用 Isolate 隔离）
```

### Dependency Notes

- **错题本状态机 requires 答题日志：** 错题不能"凭空存在"，必须有触发事件（某次作答答错）。日志表是 source of truth，ledger 视图是它的投影
- **错题抽查 enhances 错题复习：** 抽查 = 复习的轻量抽样版本，复用同一 UI 壳
- **收藏夹 enhances 乱序抽题：** 收藏的 UX 自然延伸是"只刷收藏的"——这是乱序抽题的一个 filter 参数，不是新模式
- **题库维度 conflicts 跨题库全局统计：** 学生的"高数错题"和"线代错题"语义独立；不允许在统计页混着算
- **LLM 解析 conflicts UI 阻塞：** 5 万字题库解析可能耗时 10+ 分钟；必须用 `compute()`/Isolate，且提供"后台运行 + 完成后通知"
- **本地 LLM conflicts 应用体积：** 3B 模型 4-bit 量化约 1.5-2 GB；不能与应用一同打包；首次启动按需下载，且要明确告知用户

## MVP Definition

> **MVP 目标：** "导入一个 docx，5 分钟内能用三种模式刷题"
> 任何不直接服务于此目标的功能都不进 v1。

### Launch With (v1)

- [ ] **.docx 导入** — 老师发放的主流格式；缺它 = 0 用户价值
- [ ] **本地 LLM 解析 + 进度反馈** — 核心差异化能力；缺它 = 退化为"复制粘贴工具"
- [ ] **解析结果预览 + 错误题标记/删除/编辑** — LLM 解析不可能 100%，必须给用户纠错入口
- [ ] **SQLite 持久化** — 数据不丢 = 基本信任
- [ ] **单选题 / 多选题支持** — 覆盖 90%+ 题型
- [ ] **乱序抽题模式** — 基线刷题模式
- [ ] **错题自动入错题本** — 三模式联动的核心
- [ ] **错题复习模式** — 闭环复习的承重墙
- [ ] **错题抽查模式** — 快速自测；区分"复习"和"抽查"是 UX 价值点
- [ ] **收藏 + 收藏夹** — 最小化标记能力
- [ ] **基础统计**（按题库聚合：总题数、累计正确率、错题本剩余数）— 用户判断复习效果
- [ ] **多题库管理**（导入多个 docx，每个独立）— 一学期 N 门课是常态
- [ ] **Windows + Android 打包可运行** — 交付链路打通
- [ ] **退出/崩溃后状态恢复** — 桌面 App 尊严

### Add After Validation (v1.x)

> 等用户用 MVP 真刷过几门课再加，**不预先堆**

- [ ] **.pdf 文字版导入** — 触发：用户明确反馈"题库是 PDF"
- [ ] **答题用时记录 + "慢答题"列表** — 触发：v1 统计太单薄
- [ ] **按题型/难度筛选乱序抽题** — 触发：题库 > 500 题时刷到全随机会有手感问题
- [ ] **错题本导出（csv / docx）** — 触发：用户想要"打印背"
- [ ] **题目搜索** — 触发：题库 > 1000 题
- [ ] **解析任务的暂停/恢复** — 触发：用户反馈"中途想关电脑"
- [ ] **简单的题目编辑表单（题干/选项/答案文字）** — 触发：解析预览编辑不够用

### Future Consideration (v2+)

> 这些都**不**进 v1，理由见 Anti-Features 表

- [ ] **.pdf 扫描版 / 图片内 OCR** — 复杂度爆炸（OCR 模型 + 资源管理 + 版权）
- [ ] **图片 / LaTeX 公式渲染** — 需评估 Flutter 端方案成熟度
- [ ] **填空题 / 判断题** — 判分逻辑独立工作量
- [ ] **题库导出 / 跨设备同步** — 走"导出 .db 文件"路线，不引入云
- [ ] **暗色 / 浅色主题切换** — Material 3 默认即可
- [ ] **每日打卡 / 连续天数** — Anti-feature
- [ ] **复杂统计仪表盘** — 三数字够用
- [ ] **闪卡 / 间隔重复算法** — 与错题本状态机叠加复杂度高
- [ ] **计时考试模式** — 不同产品定位
- [ ] **AI 出题** — 评测困难，质量不稳
- [ ] **多用户 / 角色** — 不同产品
- [ ] **题库分享社区** — 复杂度爆炸

### 关键竞争性观察

1. **"导入自有题库"是结构性蓝海。** Anki/Quizlet/小猿/粉笔/考研帮 都不解决"老师发的 docx 怎么变可刷题库"的问题。RedClass 的核心差异化 100% 押在这一条上。
2. **不要试图和 Anki 比记忆算法。** Anki 的 SM-2/FSRS 沉淀了 20 年。RedClass 的"乱序 + 错题本"够用；引入间隔重复会让状态机爆炸且收益边际递减（学生期末就两三周复习周期，不是终身学习）。
3. **不要试图和粉笔比内容丰富度。** 粉笔有全职教研团队，RedClass 永远比不过。RedClass 的价值是"你自己的题库"，不在"题库数量"。
4. **国内刷题 App 的"社交/打卡"是被用户骂的，不是被夸的。** 不要被"留存"指标诱导加这些功能。
