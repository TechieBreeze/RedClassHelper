# Phase 03 — UI Design Contract

> Visual and interaction contract for the Desktop LLM Integration & Parse Quality phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker. Extends Phase 2's import pipeline with LLM parsing, model management, and parse-source traceability.

---

## Phase Scope (UI-Relevant)

This phase layers on-device LLM parsing onto the Phase 2 import pipeline, adds a model management page, and makes parse-source identification visible to the user. All LLM UI surfaces are desktop-only (gated on `Platform.isWindows || Platform.isLinux`).

**Owned requirements** (per REQUIREMENTS.md traceability):
- **IMP-03** — Desktop app invokes local on-device small LLM to parse raw text into structured questions
- **IMP-04** — Parse progress shown with failure reasons; retry on failure (extended with LLM sub-phase + per-question fallback)

**What this phase also delivers:**
- Import-mode selection dialog — appears after file selection, before parse begins (D-01)
- Model management page — Settings entry, catalog of 3 preset tiers, download-on-demand (D-03, D-04, D-05, D-06)
- Extended progress screen with LLM parsing sub-phase (D-07)
- Auto-confirmed LLM results bypassing manual review (D-08)
- Per-question source identification on summary screen: LLM / 启发式 / 兜底 (D-09)

**What this phase does NOT include:**
- Quiz loop (Phase 4)
- JSON export/import (Phase 5)
- Full Settings screen with all entries (Phase 6)
- Diagnostic pack / parse_log viewer (Phase 6)
- First-run model download onboarding (Phase 6)
- FFI binding for llama.cpp (Plan 03-07/03-08 — research spike only)

**UI surface delivered in Phase 3:**
1. **Parser choice dialog** — modal dialog after file selection, 2 options
2. **Settings screen** (minimal) — entry point for model management (full Settings in Phase 6)
3. **Model management page** (`/settings/models`) — catalog, download, custom model
4. **Extended ImportProgressScreen** — LLM parsing sub-phase with question-by-question progress
5. **Updated ImportPreviewScreen** — LLM results auto-confirmed; heuristic results manual review; parse-source badge per question
6. **Extended ImportSummaryScreen** — per-question source identification (LLM/启发式/兜底)

## Color

Phase 1/2 color system carries forward. Phase 3 semantic additions:

| Role | Light Value | Usage (Phase 3 additions) |
|------|-------------|---------------------------|
| Primary (Accent 10%) | Phase 1 seed + dynamic_color | Download button; download progress bar fill; active dialog option border; "已安装" installed badge |
| Secondary (30%) | Phase 1 derived | Model card backgrounds; dialog option card backgrounds; parser choice dialog surface |
| Surface (60%) | Phase 1 surface | All screen backgrounds; model management page |
| Error/Destructive | `Color(0xFFB3261E)` (light) | Delete model button; Sha256 verification failed; download failed |
| **LLM source** | `Colors.teal` shade (via `colorScheme.tertiary`) | "LLM 解析" source badge on summary screen |
| **Heuristic source** | `colorScheme.secondary` | "启发式解析" source badge on summary screen |
| **Fallback source** | `Colors.amber.shade700` | "兜底解析" (LLM failed → heuristic fallback) source badge |
| **Tier: Recommended** | `colorScheme.primary` tone | "推荐" tier badge on model card |
| **Tier: Fast** | `Colors.green` tone (via `colorScheme.secondary`) | "快速" tier badge on model card |
| **Tier: Experimental** | `Colors.deepOrange` tone | "实验" tier badge on model card |
| **Download success** | `Colors.green.shade600` | Sha256 verified checkmark; "下载完成" status |
| **Download warning** | `Colors.orange.shade700` | Incomplete download; resume prompt |

**Accent (primary) reserved for Phase 3:**
- "下载" primary button on model cards (desktop-only)
- Download progress bar fill
- Selected parser choice dialog option border (2dp)
- "高精度解析（LLM）" option icon tint when selected

**Accent NOT used for:**
- Model card backgrounds (use `surfaceContainerHighest`)
- Tier badges (each has its own semantic color, distinct from primary accent)
- Source identification badges on summary (each has its own semantic color)
- Dialog background (use `surfaceContainerHigh`)

**New semantic color rationale:**
Phase 3 introduces "parse source" as a first-class visual concept. Three distinct source badges (LLM / 启发式 / 兜底) need three visually distinct, non-primary colors so users can scan the summary screen and immediately understand parse quality distribution. These colors are semantic — they encode meaning (model-parsed = trusted teal, rule-based = neutral secondary, fallback = caution amber) — not decorative.

```
Disabled state (no model installed):
  Option 2 card: dimmed at 50% opacity.
  Below description: "需要先下载模型" (12px, error color).
  Below that: "前往设置 → 模型管理下载" (12px, secondary color, tappable link).

Selected state:
  Selected card: 2dp primary border + subtle primary container background tint.
```

**Interaction rules:**
- Tap an option card → selects it (visual: 2dp primary border, 150ms `AnimatedContainer` transition)
- Tap the other option → switches selection
- Tap "取消" → dismisses dialog, returns to ImportScreen, no parse started
- Tap outside dialog → no-op (modal is not backdrop-dismissible)
- If LLM selected but no model installed → option card is tappable but shows disabled sub-state; "取消" is the only exit
- Decision is NOT persisted — dialog appears fresh on every import (D-01)

## Claude's Discretion — Design Rationale

Areas where CONTEXT.md delegated design decisions to the researcher:

### 1. Model Management Page Layout
**Decision:** Three-section vertical scroll: "已安装模型" (top, collapsible when empty), "推荐模型" (middle, always visible catalog), "自定义模型" (bottom, add button + user-added models). The installed section appears first to prioritize "what do I have" over "what can I get." The catalog section is always visible so first-time users see the recommendation immediately. Custom models are at the bottom because they are the least common path.
**Rationale:** Follows Material 3 list-detail pattern. Three sections provide clear information hierarchy: status (installed) → discovery (catalog) → extension (custom).

### 2. Parser Choice Dialog Design
**Decision:** Two tappable card options (not radio buttons + confirm). Selected card shows 2dp primary border. Single-tap to select — no "确认" button needed. "取消" is the only action button.
**Rationale:** Reduces friction — one tap to choose and start. The dialog is a binary choice with clear trade-offs; a radio-button pattern would add an unnecessary confirmation step. The inline disabled state for LLM (when no model) avoids an error-after-tap flow.

### 3. Source Identification Badges
**Decision:** Three distinct semantic colors: teal (LLM = trusted), secondary (heuristic = baseline), amber (fallback = caution). Badges appear on both preview and summary screens. On summary, an aggregate "解析来源" section provides a bird's-eye view.
**Rationale:** Color-coding leverages pre-attentive processing — users can scan the summary and instantly gauge parse quality distribution. The amber fallback color is intentionally distinct from error (red) because fallback is a successful recovery, not a failure.

### 4. Model Download Progress
**Decision:** In-card progress bar (not a separate screen or modal). Speed display updates every 500ms. "取消" button alongside progress for abort. Sha256 verification is automatic post-download with a brief "校验中…" state.
**Rationale:** In-card progress keeps the model catalog browsable during download. Users can read about other models while one downloads. Separate-screen progress would block exploration.

### 5. Minimal Settings Screen
**Decision:** Only the model management entry is rendered in Phase 3. The settings screen itself is a simple `ListView` with one `ListTile`. Other entries (theme, font size, diagnostics, about) are reserved for Phase 6 and are NOT rendered as disabled placeholders — they are simply absent.
**Rationale:** Phase 3's scope is LLM integration, not settings completeness. Rendering disabled placeholder entries would create visual noise and false expectations. The settings screen grows organically in Phase 6.
